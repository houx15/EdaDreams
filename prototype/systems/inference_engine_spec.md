# 推理引擎规格书 — Case 00 + Case 01

> EdaDreams · 系统设计文档 · v1 · 2026-04-13
>
> 本文件供 coder 实现推理引擎。定义：每一步推理的 context 关键词权重、
> 输出生成规则、解锁触发条件、以及不同 context 质量对应的输出变体。

---

## 一、推理引擎总体机制

### 核心公式

```
输出质量 = f(context 中命中的关键词数 × 权重)
```

**不是 LLM 实时生成。** 每一步的推理输出是预写好的模板，根据 context 命中情况拼接/裁剪。这样保证：
1. 输出可控、可测试
2. 不依赖运行时 AI 调用
3. 玩家每次推理都能得到确定性的反馈

### 匹配规则

- 扫描 context 中所有 block 的文本内容
- 对每个关键词做**模糊匹配**（包含即命中，不要求精确位置）
- 同一关键词在多个 block 中出现只算一次命中
- 中文匹配忽略空格和标点

### 输出等级

每一步推理有三个输出等级：

| 等级 | 条件 | 效果 |
|------|------|------|
| **A — 完整** | 命中所有 ★★★ 关键词 + 至少 1 个 ★★ 关键词 | 生成完整分析，包含触发词，解锁下一步 |
| **B — 部分** | 命中至少 1 个 ★★★ 关键词，但缺少其他 | 生成不完整分析，缺失部分标为 `[数据不足]`，可能不含触发词 |
| **C — 失败** | 未命中任何 ★★★ 关键词 | 生成简短提示："当前上下文信息不足，无法得出有效结论。请补充相关证据。" |

**特殊情况：** Context 完全为空 → 返回 `上下文为空，无可推理内容。`

### 解锁判定

解锁下一份证据的条件是**推理输出文本**中包含指定触发词（不是 context 中包含）。只有 A 级输出保证包含触发词。B 级输出有概率包含（约 50%——实现时可以做成：如果命中了与触发词直接相关的 ★★★ 关键词，则 B 级也包含触发词）。

Case 00 只有一次推理（B6 阶段），输入是 `briefing_001.eml`。

| 关键词 | 权重 | 含义 |
|--------|------|------|

| `钓鱼` 或 `phishing` | ★★★ | 事件类型 |

| `陆明远` | ★ | 对接人 |
| `1,183` 或 `1183` | ★★ | 事件规模 — 邮件数 |
| `47` （在"客户"或"受害"附近） | ★★ | 事件规模 — 凭证泄露数 |
| `11` （在"转账"或"账户"附近） | ★ | 事件规模 — 转账数 |
| `384,000` 或 `384000` 或 `38万` | ★★ | 事件规模 — 金额 |
| `钓鱼域名` 或 `域名` | ★ | 证据类型 |
| `追踪` 或 `源头` | ★ | 目标 |

### 输出模板

**A 级（命中：案件编号 + 钓鱼 + 至少 1 个规模数字）**
→ 使用 `analysis_001.rpt` 全文（见 case00_boot.md B6 节）

**B 级（命中：钓鱼，但缺案件编号或规模）**
```
案件: [未指定]
类型: 初步分析

本案涉及钓鱼攻击，但当前上下文缺少关键信息。

已识别:
  · 事件类型: 凭证钓鱼攻击
  [以下根据命中情况动态填充或标为"数据不足"]
  · 事件规模: [数据不足]
  · 委托方: [数据不足]
  · 已有证据: [数据不足]

建议补充案件编号、规模数据和证据清单后重新推理。
```

**C 级** → 通用失败文本

### 噪音词（出现在 context 中会稀释输出）

以下内容如果出现在 context 中但**不伴随**关键词，会在 B 级输出中插入干扰段落：

| 噪音 | 稀释效果 |
|------|---------|
| `祝工作顺利` / `欢迎上线` | 输出末尾多一段无用的"感谢委托方信任"客套 |
| `ops@eda.sec` / `eda@eda.sec` | 输出中多花篇幅分析邮件头，减少对案件本身的分析 |
| `Tier-1` / `运营方` | 输出多一段关于权限说明的冗余文字 |

**设计意图：** 噪音不阻止推理成功，但让输出变啰嗦、重点不突出。玩家反复推理后会悟出"原来塞那些没用的东西会让 Eda 跑偏"。

**⚠️ 特殊解锁条件:**

本步触发词需要**双重匹配**:
1. 输出包含 `刘哲`（来自本步证据）
2. 输出包含 `受害` / `客户名单` / `受影响`（来自 context 中残留的 briefing 信息）

**实现逻辑：** 推理引擎在生成 Step 06 输出时，额外检查 context 中是否包含 `受影响客户` / `客户名单` / `受害者` 等词。如果有，A 级输出的"三、关键疑问"段落会包含"检查名单中是否有刘哲"——从而触发解锁。如果 context 中没有这些词，即使其他关键词全部命中，输出也不会产生这个关联，解锁被阻断。

**这是全案唯一一个需要玩家保持旧信息在 context 中的步骤。** 如果玩家在 Step 05–06 之间清掉了 briefing 的所有内容，就会卡在这里。他们需要回头重新把 briefing 中"受影响客户名单"相关的句子拖回 context。

| `384,200` 或 `384200` | ★★ | 总金额 |
| `3301` 或 `8847` | ★★ | 资金链 |
| `01-05` 或 `准备期` | ★ | 时间线 |
| `ATM` 或 `加密货币` | ★ | 洗钱手段 |

**输出逻辑:** 最终报告根据命中的关键词数量决定完整度。每个大节（案件概述、攻击者、攻击链、证据清单、结论）对应不同关键词组。缺哪组，报告中对应章节标为 `[信息不足，未纳入报告]`。

**A 级（命中 5+ 组）:** 完整五章报告（见剧本 Step 12）
**B 级（命中 3–4 组）:** 三章报告，缺失部分有 `[数据不足]` 标记
**C 级（命中 0–2 组）:** 仅生成一段概述 + "本报告信息不完整，无法作为正式分析提交"

**结案条件:** A 级报告生成后，案件自动结案。B/C 级不触发结案——玩家需要调整 context 重试。

## 四、噪音稀释规则（全局）

当 context 中包含以下类型的内容时，推理输出会被"稀释"——不是变错，而是变啰嗦、重点不突出：

| 噪音类型 | 稀释效果 |
|----------|---------|
| 完整的证据原文（而非摘要/关键句） | 输出中多出"原始材料显示…"的引述段落，浪费篇幅 |
| 之前步骤的完整推理输出 | 输出开头多一段"基于前次分析…"的回顾，但不影响新结论 |
| 与当前步骤无关的证据 | 输出中多一段"注：上下文中包含与当前分析无关的材料"提示 |
| 重复的信息（同一事实出现在多个 block 中） | 输出中对该事实过度展开 |

**稀释的实现建议：** 在 A 级输出模板的基础上，检测噪音关键词，如果命中，在指定位置插入预写的冗余段落。玩家读到这些冗余段落会意识到"我塞多了"。

## 五、Context 容量规则

### 百分比计算

每份证据和每段推理输出有一个预设的 token 数（用字符数近似，1 中文字 ≈ 2 token）。context 面板显示的百分比 = 当前总 token / 最大 token。

### 各文件预设 token 占比（Case 01，context 上限 = 100%）

| 文件 | 预估字符数 | 占比（如果整篇拖入） |
|------|-----------|---------------------|
| briefing_001.eml（Case 00 残留） | ~800 | ~8% |
| analysis_001.rpt（Case 00 残留） | ~500 | ~5% |
| 01_phishing_sample.eml | ~900 | ~9% |
| 02_domain_registration.txt | ~600 | ~6% |
| 03_email_account.txt | ~800 | ~8% |
| 04_phone_records.txt | ~1,200 | ~12% |
| 05_phone_lookup.txt | ~400 | ~4% |
| 06_company_registration.txt | ~800 | ~8% |
| 07_affected_clients.txt | ~2,000 | ~20% |
| 08_liuzhe_account.txt | ~1,000 | ~10% |
| 09_fund_flow.txt | ~1,200 | ~12% |
| 10_account_b.txt | ~500 | ~5% |
| 11_timeline.txt | ~1,500 | ~15% |
| 每步推理输出（平均） | ~400 | ~4% |

**总证据量 ≈ 200%。** 这意味着即使没有任何推理输出的累积，玩家也不可能同时放入所有证据。从 Step 06 开始，主动管理 context 是必须的。

### 容量条颜色

| 占用 | 颜色 | 含义 |
|------|------|------|
| 0–50% | 蓝色 | 安全 |
| 50–80% | 橙色 | 注意 |
| 80–100% | 红色 | 紧张 |
| >100% | 红色闪烁 + `CONTEXT OVERFLOW` | 溢出，禁止添加 |

## 六、实现优先级建议

1. **先做关键词匹配 + A 级输出** — 确保 12 步链能走通
2. **再做 B 级输出** — 确保玩家犯错时有合理反馈
3. **最后做噪音稀释** — 锦上添花，不影响核心流程
4. **C 级输出用统一模板** — 最低优先级
