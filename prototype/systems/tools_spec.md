# 工具系统规格书（设计意图文档）

> EdaDreams · 系统设计文档 · **v2** · 2026-04-14
>
> **定位：** 本文件描述工具系统的设计意图和交互原则（为什么这么设计）。
> **实现规格**（具体的输入→输出映射）见 `case01_data/tools_query_map.json` 和 `search_engine.json`。
>
> 本文件定义玩家可使用的所有工具。
> 工具分三类：**网页**（浏览器中打开，需要从备忘找网址和账号）、
> **电话**（拨打 NPC）、**数据库**（需要授权后登录查询）。
> 玩家的操作全部手动完成。后续 case 中，部分工具可以委托给 Eda 自动执行。

---

## 设计原则

1. **玩家像用真电脑一样操作。** 每个工具就是一个 app/网页/终端。打开、输入、等结果、翻结果、判断有没有用。
2. **搜错了就搜错了。** 输入无关关键词 → 返回无结果或一堆噪音。没有提示"你是不是要搜xxx"。
3. **结果里有噪音。** 正确的搜索也可能返回多条结果，玩家要自己判断哪条有用。
4. **不同信息去不同工具找。** 查域名用域名查询，查公司用工商系统，要银行材料找对接人要。玩家得想"这个信息应该去哪找"。
5. **工具从 Case 01 起就可以手动使用，🔒 锁的是 Eda 自动使用的权限。**

---

## 工具分类

工具不在侧边栏直接列出。玩家通过阅读 `investigation_resources.note`（briefing 附件）获知可用工具的网址、电话号码和登录方式。

### 分类

| 类型 | 工具 | 交互方式 | 如何找到 |
|------|------|---------|---------|
| 🌐 网页 | 域名查询（WHOIS） | 浏览器 tab，输入域名查询 | 备忘中有网址+账号 |
| 🌐 网页 | 工商信息查询 | 浏览器 tab，搜企业名/法人 | 备忘中有网址，公开无需登录 |
| 🌐 网页 | 号码归属查询 | 浏览器 tab，输入号码查询 | 备忘中有网址+账号 |
| 🌐 网页 | 网页搜索引擎 | 浏览器 tab，自由搜索 | 侧边栏有入口（通用工具） |
| 📞 电话 | 联络（陆明远） | 拨号界面，语音对话 | briefing 里有电话号码 |
| 💾 数据库 | 银行调查系统 | 终端界面，结构化查询 | 需电话陆明远申请授权 |
| 💾 数据库 | 通信记录系统 | 终端界面，按号码查询 | 需电话陆明远逐号申请 |

### 发现机制

**没有工具清单。所有工具通过搜索引擎发现。**

**网页工具：** 玩家在搜索引擎里搜"域名查询"、"WHOIS"、"工商信息查询"、"号码归属"等关键词 → 搜索结果中出现对应网站链接 → 点击打开。部分网站需要注册/登录——玩家可以尝试用 Eda 的调查员身份注册，或搜索是否有公开免费的查询入口。

**电话（陆明远）：** Briefing 里只写了"对接人: 陆明远 · 高级安全工程师"，没有电话号码。玩家需要在搜索引擎搜"东海银行 陆明远"或"东海银行 网络安全响应中心 联系方式"→ 搜索结果中出现联系电话 → 玩家拨打。

**数据库：** 玩家打电话找到陆明远后，在通话中申请银行系统权限。陆明远确认后告诉玩家登录地址和临时账号。通信记录同理——通话中申请，陆明远协调运营商后回电告知结果。

## 工具 1: 网页检索

### 界面

在中间阅读区打开一个 tab，界面类似简化版搜索引擎：

```
┌──────────────────────────────────────────────┐
│  🔍 网页检索                                  │
│                                              │
│  ┌────────────────────────────────┐ [搜索]   │
│  │                                │          │
│  └────────────────────────────────┘          │
│                                              │
│  搜索结果:                                    │
│  ─────────────────────────────────           │
│  （输入关键词后显示结果列表）                    │
│                                              │
└──────────────────────────────────────────────┘
```

### 交互

- 玩家在搜索框输入任意文本，点击「搜索」或回车
- 返回 0–10 条结果，每条显示标题 + 摘要 + 来源
- 点击某条结果 → 展开显示完整内容（可选中拖拽到 context）

### 搜索结果库

搜索引擎使用**模糊关键词匹配**。每个预设查询有一组匹配关键词和对应结果。

**Case 01 预设结果：**

| 搜索关键词（任一命中即触发） | 结果 |
|---|---|
| `donghai-verify` / `donghai-secure` / `donghai-safety` | 1 条有用结果：域名安全论坛帖子"有人遇到 donghai-verify.cn 钓鱼吗？" + 2 条噪音 |
| `东海银行 钓鱼` / `东海银行 诈骗` | 3 条新闻摘要（其中 1 条提到"近日东海银行客户遭钓鱼攻击"）+ 2 条旧新闻噪音 |
| `瑞泽信息技术` / `瑞泽信息` | 1 条工商信息摘要 + 1 条噪音（另一个叫"瑞泽"的公司） |
| `刘哲 深圳` | 2 条噪音（同名不同人的社交媒体） |
| `陈芳 海口` | 3 条噪音（同名不同人） |
| `vps996` | 1 条：该服务器商的官网简介（廉价 VPS 供应商） |
| 其他任意关键词 | 0–3 条随机噪音结果 |

**噪音结果示例：**

```
搜索: 东海银行 钓鱼

结果 1 ──────────────────────────
东海银行回应客户钓鱼攻击事件
来源: 南方财经网 · 2041-03-17
"东海商业银行今日发布声明，确认近日有客户遭遇钓鱼邮件攻击，
目前已报警处理并冻结相关可疑账户……"
→ 有用。确认事件公开。

结果 2 ──────────────────────────
东海信用社储户被骗 12 万元
来源: 海都报 · 2039-08-22
"东海农村信用社一储户称接到自称银行客服的电话后……"
→ 噪音。旧新闻，不同机构。

结果 3 ──────────────────────────
钓鱼爱好者论坛 — 东海海钓攻略
来源: 钓友之家 · 2041-02-10
"东海春季最佳钓点推荐……"
→ 噪音。字面意义的钓鱼。
```

### 设计要点

- **网页检索是最泛的工具，什么都能搜但结果最杂。** 它主要用于：玩家不知道用什么专门工具时的"万能后备"，或者想看看某个线索在公开互联网上有没有信息。
- **Case 01 中它不是主要工具。** 大部分关键信息在专用工具（域名查询、工商信息等）里更容易找到。但如果玩家先搜网页，也能获得一些方向性线索。
- **噪音结果要看着像真的。** 标题、来源、日期都要可信。玩家需要自己判断哪条有用。

## 工具 2: 域名查询

### 界面

类似 WHOIS 查询网站：

```
┌──────────────────────────────────────────────┐
│  域名备案/WHOIS 查询                           │
│                                              │
│  域名: ┌──────────────────────┐ [查询]       │
│        │                      │              │
│        └──────────────────────┘              │
│                                              │
│  查询结果:                                    │
│  ─────────────────────────────               │
│                                              │
└──────────────────────────────────────────────┘
```

### 交互

- 输入一个域名（如 `donghai-verify.cn`），点查询
- 如果该域名在预设库中 → 返回完整注册信息
- 如果不在预设库中 → 返回「未查询到该域名的备案信息」

### 预设域名库

| 输入域名 | 返回内容 |
|---------|---------|
| `donghai-verify.cn` | 完整注册信息（见 `content/case01/02_domain_registration.txt` 中对应段落） |
| `donghai-secure.cn` | 完整注册信息 |
| `donghai-safety.cn` | 完整注册信息 |
| `donghai-bank.com` | 「该域名注册人: 东海商业银行股份有限公司 · 2018-03-01」（正规域名，让玩家确认对比） |
| 其他 | 「未查询到该域名的备案信息」 |

### 设计要点

- **这是玩家最可能第一个使用的专用工具。** 他们在钓鱼邮件里看到 `donghai-verify.cn`，自然想到"查查这个域名"。
- 三个钓鱼域名都能查到，而且结果里都显示同一个注册邮箱 `lz_tech@163.com` 和注册人"刘哲"——如果玩家查了多个域名，这个重合自然浮现。
- 如果玩家只查了一个域名也够用——一个就能拿到邮箱线索。

## 工具 3: 工商信息

### 界面

类似国家企业信用信息公示系统：

```
┌──────────────────────────────────────────────┐
│  国家企业信用信息公示系统                        │
│                                              │
│  企业名称/统一信用代码/法人姓名:                  │
│  ┌────────────────────────────────┐ [查询]   │
│  │                                │          │
│  └────────────────────────────────┘          │
│                                              │
│  查询结果:                                    │
│  ─────────────────────────────               │
│                                              │
└──────────────────────────────────────────────┘
```

### 交互

- 支持按企业名称、信用代码、法人姓名搜索
- 可能返回多条结果（同名公司/同名法人），玩家自行判断

### 预设库

| 搜索词 | 返回结果 |
|--------|---------|
| `瑞泽信息技术` / `瑞泽信息` / `瑞泽` | 2 条：① 瑞泽信息技术有限公司（深圳，刘哲）✓ ② 瑞泽网络科技有限公司（杭州，张某，2038 年成立）← 噪音 |
| `刘哲`（按法人搜索） | 3 条：① 瑞泽信息技术有限公司 ✓ ② 刘哲餐饮管理有限公司（成都）← 噪音 ③ 深圳刘哲商贸（已注销）← 噪音 |
| `东海银行` / `东海商业银行` | 1 条：东海商业银行股份有限公司（上海，正规大企业）|
| 其他 | 「未查询到相关企业信息」或 1–2 条无关噪音 |

### 设计要点

- **玩家可能通过两种路径到达这里：** ① 在号码归属查到"瑞泽信息技术"后来搜 → 按企业名搜。② 在域名注册看到"刘哲"后来搜 → 按法人名搜。两条路都通。
- 按"刘哲"搜时返回多条结果，玩家得根据地址（深圳南山）和时间来判断哪个是目标。

| `0755-8832-6617` | 归属地: 广东深圳南山 · 类型: 固定电话 · 登记单位: 瑞泽信息技术有限公司（公开的企业登记电话可查到单位名） |

- 手机号只能查到归属地和运营商，要知道"谁的号"必须走正式渠道（联络陆明远申请调取）。

## 工具 5: 银行系统 🔐

### 解锁方式

初始状态：点击显示「本工具需要委托方授权。请联络对接人申请开通。」

玩家通过「联络」工具给陆明远发消息，申请银行系统查询权限。陆明远回复确认后，工具解锁。

### 解锁后界面

```
┌──────────────────────────────────────────────┐
│  东海银行 · 安全调查协作系统                     │
│  授权人: 陆明远 · 权限级别: 案件关联查询          │
│                                              │
│  查询类型: [客户信息 ▾]                         │
│  查询条件: ┌──────────────────┐ [查询]        │
│           │                  │               │
│           └──────────────────┘               │
│                                              │
│  查询类型可选:                                  │
│    · 客户信息（按姓名/身份证/账号）               │
│    · 交易流水（按账号 + 时间范围）                │
│    · 受影响客户名单                              │
│                                              │
└──────────────────────────────────────────────┘
```

### 预设库

| 查询类型 | 输入 | 返回 |
|---------|------|------|
| 客户信息 | `刘哲` / 身份证号 / 账号 3301 | `08_liuzhe_account.txt` 的账户信息部分 |
| 客户信息 | `陈芳` / 账号 8847 | `10_account_b.txt` 内容 |
| 交易流水 | 账号 3301 + 日期范围 | `08_liuzhe_account.txt` 的流水部分 |
| 交易流水 | 账号 8847 + 日期范围 | 资金流出明细（ATM/支付宝/加密货币——来自 `09_fund_flow.txt`） |
| 受影响客户名单 | （不需要输入，直接点击） | `07_affected_clients.txt` 内容 |
| 其他 | 任意名字 | 「未查询到该客户信息」或「该客户与本案无关联」 |

### 设计要点

- **银行系统是信息量最大的工具，但它需要授权才能用。** 玩家得主动想到"我需要查银行资料"然后去联络陆明远要权限。
- **"受影响客户名单"是一个直接选项**——但如果玩家不知道要看名单，他们可能会错过它而只查"刘哲"的个人信息。名单里的"啊哈时刻"（刘哲出现在名单中）需要玩家自己想到去看名单。
- **陈芳的信息需要玩家在追踪资金流向后才知道要查**——这是自然的：先看刘哲流水 → 发现资金去向 8847 → 查 8847 账号 → 发现陈芳。

## 工具 6: 通信记录 🔐

### 解锁方式

## 工具: 电话（联络）

### 界面

类似桌面电话 / VoIP 软件：

```
┌──────────────────────────────────────────────┐
│  ☎ 通话                                      │
│                                              │
│  拨号: ┌──────────────────┐ [拨打]           │
│        │ 0755-8800-1234   │                  │
│        └──────────────────┘                  │
│                                              │
│  ─── 通话进行中 ───                            │
│  对方: 陆明远 · 东海银行 · 高级安全工程师         │
│  时长: 00:12                                  │
│                                              │
│  [陆明远] 你好，Eda，有什么需要？               │
│                                              │
│  ┌────────────────────────────────┐ [说]     │
│  │ 输入你要说的话...               │          │
│  └────────────────────────────────┘          │
│                                              │
│                               [挂断]         │
└──────────────────────────────────────────────┘
```

### 交互

- 玩家输入电话号码拨打（或从备忘/briefing 中点击号码直拨）
- 接通后进入文字化的"通话"——玩家打字说话，NPC 打字回复
- 视觉上模拟电话：有通话时长计时、音波动画、对方身份显示
- 系统对玩家输入做**关键词匹配**，陆明远回复
- 回复延迟 1–3 秒（模拟对方思考）
- 玩家可以随时挂断

### 陆明远的回复预设

| 玩家消息中的关键词 | 陆明远的回复 |
|---|---|
| `客户名单` / `受影响` / `受害者` / `名单` | "好的，我把受影响客户名单的查询权限开给你了。你去银行系统里面查就行。" → **银行系统中"受影响客户名单"功能解锁** |
| `银行` + (`账户` / `流水` / `查询` / `权限`) | "可以，我帮你开通银行调查系统的权限。你现在可以查询与本案相关的客户信息和交易流水了。" → **银行系统工具整体解锁** |
| `通信记录` / `手机` / `电话` + (`调取` / `查询` / `记录`) | "需要走运营商调取流程。你要查谁的号码？告诉我号码，我帮你提交申请。" |
| `138-2771-9403` / `13827719403` | "好，138-2771-9403 的通信记录已经调到了。你去通信记录里查吧。" → **该号码在通信记录工具中解锁** |
| `189-7654-3210` / `18976543210` | "这个号码我也帮你调了，去看看。" → **该号码在通信记录工具中解锁** |
| `邮箱` / `163` / `lz_tech` | "邮箱账号的信息我帮你向网易那边调取，稍等。" → 5 秒后："调到了，我把结果放到你的证据材料里了。" → **邮箱信息出现在银行系统或证据区** |
| `进展` / `怎么样了` / `更新` | "一切由你主导，有发现随时告诉我。我这边配合你。" |
| 其他无法匹配的消息 | "收到。如果你需要调取什么材料或者权限，具体告诉我就行。" |

### 设计要点

- **陆明远是唯一的 NPC，他是"资源接口"而不是"信息提供者"。** 他不会主动给线索或方向。他只是帮你开权限、帮你调资料。
- **玩家必须自己想到要什么，然后找陆明远要。** 比如，看到刘哲的注册手机号后，玩家得自己想到"我需要查这个手机号的通信记录"→ 去联络陆明远 → 告诉他号码 → 他帮忙调取 → 通信记录工具里可以查了。
- **邮箱账号信息是个特殊情况。** 没有单独的"邮箱查询"工具（因为普通人确实没有这种工具）。玩家需要联络陆明远说"帮我查查 lz_tech@163.com 这个邮箱"，由陆明远走官方渠道调取。
- **陆明远的回复风格：** 简短、专业、配合。不废话，不给方向性建议。他是一个工具人（literally）。

## 新的调查流程概览

```
briefing_001.eml（唯一主动投递的文件）
       │
       ├─ 玩家在邮件里看到 donghai-verify.cn
       │  └─ 玩家打开「域名查询」→ 输入域名 → 得到注册人刘哲 + lz_tech@163.com
       │
       ├─ 玩家拿到邮箱 lz_tech@163.com
       │  └─ 哪里查邮箱？没有专用工具 → 联络陆明远"帮我查这个邮箱"
       │     └─ 陆明远调取 → 得到手机号 138-2771-9403 + 深圳 IP
       │
       ├─ 玩家拿到手机号
       │  ├─ 先去「号码查询」→ 只能查到归属地深圳
       │  └─ 要通话记录 → 联络陆明远申请调取 → 通信记录里查到频繁联络的固话
       │
       ├─ 玩家拿到固话 0755-8832-6617
       │  └─ 去「号码查询」→ 查到登记单位"瑞泽信息技术有限公司"

       │  └─ 去「工商信息」→ 查到法人刘哲 + 壳公司特征
       │
       ├─ 玩家此时应该想到"查查银行那边有没有刘哲的信息"
       │  └─ 联络陆明远要银行系统权限 → 解锁
       │     ├─ 查"受影响客户名单" → 发现刘哲在名单里（啊哈！）
       │     ├─ 查刘哲账户流水 → 发现资金全转向 8847
       │     └─ 查账户 8847 → 发现持有人陈芳 + 手机号 189-7654-3210
       │
       ├─ 玩家发现 189-7654-3210（如果他们还记得通话记录里出现过）
       │  └─ 可以回通信记录确认 / 联络陆明远调取该号码的记录

       └─ 最终报告
```

**关键变化：** 这不再是线性的 12 步链。玩家可以按任意顺序探索。有些路径更高效，有些会绕远路，但都能最终到达终点。推理引擎的作用是"帮玩家整理已有信息并提示方向"，不是"触发下一份文件的投递"。

## 推理引擎的新角色

推理的输入输出机制不变（context curation → 推理按钮 → 分析输出）。但输出的作用变了：

**旧设计：** 输出包含触发词 → 系统自动投递下一份文件
**新设计：** 输出提供方向性建议 → 玩家自己决定去哪个工具查

输出示例（放入域名注册信息后推理）：

```
分析阶段: 域名注册溯源

一、发现
  三个钓鱼域名均由同一人注册:
  注册邮箱 lz_tech@163.com，注册人刘哲。

二、建议调查方向
  1. 查询 lz_tech@163.com 的账号详细信息
  2. 查询"刘哲"的工商登记记录
  3. 检查域名 DNS 指向的服务器 vps996.com
```

注意"建议调查方向"只是提示——不是任务列表。玩家可以选择按它走，也可以按自己的直觉走别的路。如果玩家 context 不够好，推理输出里的建议会更模糊或缺失。

## 工具的委托演化（预告，Case 02+ 实现）

| Case 01 | Case 02+ |
|---------|----------|
| 玩家手动打开域名查询，输入域名 | Eda 自动识别文本中的域名并查询 |
| 玩家手动给陆明远发消息要权限 | Eda 自动与对接人沟通获取权限 |

这就是 delegation 的完整弧线：Case 01 的痛苦手动操作 → 后续 case 逐步交给 Eda。
