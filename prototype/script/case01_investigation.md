## 整体结构

本案共 **12 步推理**，串成一条完整的调查链。玩家从一封钓鱼邮件出发，沿着域名 → 邮箱 → 手机号 → 公司 → 嫌疑人 → 资金 → 同伙的路径追踪，最终重建整条攻击链并出具报告。

**核心节奏：**
- 步骤 1–5：顺畅的线索追踪。玩家学会"读证据 → 挑关键信息进 context → 推理 → 拿到新文件"的循环。Context 逐渐变满。
- 步骤 6–8：转折区。刘哲同时出现在公司法人和受害者名单里——这是全案的"啊哈时刻"。同时 context 已经满了，玩家被迫开始手动删旧 block、只保留结论。
- 步骤 9–11：收网。资金追踪、同伙浮出、时间轴拼合。玩家此时在反复做"删旧的、留关键的、推理、再删"的操作——非常累，但有收获感。
- 步骤 12：最终报告。玩家把全案关键结论拼进 context，一次推理生成完整分析报告，结案。

**教学目标（不通过文字，通过体验）：**
- "推理引擎的输出质量完全取决于我塞进去的东西"（延续 Case 00）
- "Context 是有限的，旧的信息要为新的腾地方"（为 Case 02 的 compaction 委托埋种子）
- "我需要自己判断什么是关键信息"——这个游戏考的是阅读理解和信息筛选，不是操作速度

---

## 通贯规则（Case 01 特有）

- **证据逐步解锁**：每一步推理的输出中包含一个关键词（人名、域名、号码等），系统检测到该关键词后，自动向"证据材料"列表推送下一份文件。玩家看到的效果是侧边栏 badge 数字 +1，状态栏滚出一行日志。
- **推理输出也占 context**：上一步推理的 output 会自动留在 context 里（除非玩家手动删掉）。这意味着每推理一次，可用空间就少一些。
- **没有"正确答案"的强校验**：玩家可以在任意步骤用任意 context 内容推理。只有当输出里包含特定关键词时，才会触发下一份证据。如果玩家 context 太差、推理输出里没提到那个词，证据就不会解锁——玩家需要调整 context 再试。
- **不提示玩家"你漏了什么"**：系统不会说"请把域名加入上下文"。玩家只能通过推理输出的质量来间接判断自己的 context 是否到位。
- **Case 01 的 context 上限设为 100%**（相对于 Case 00 的 30%，context 窗口更大了——但证据量也大得多）。

## Step 01 — 钓鱼邮件样本

### 触发

Case 00 结束后（玩家点击 `[ 接受案件 ]`），侧边栏「证据材料」里出现第一份新文件：

```
phishing_sample.eml
case_001 · 证据 #01 · 8.2 KB
```

状态栏滚出：`输入流 · 证据材料已送达 · phishing_sample.eml`

### 玩家动作

点击打开，在中间阅读区阅读全文（完整文本见 `content/case01/01_phishing_sample.eml`）。

### 证据内容要点

这是一封完整的钓鱼邮件，包括：
- **发件人**：显示为 `service@donghai-bank.com`（伪造的）
- **实际发件地址**（在邮件头"详细信息"折叠区里）：`noreply@donghai-verify.cn`
- **邮件正文**：恐吓语气，"您的账户存在异常，请在24小时内点击以下链接验证身份，否则账户将被冻结"
- **钓鱼链接**：`https://www.donghai-verify.cn/auth/login`
- **邮件底部**：仿冒的银行落款，甚至有一行"本邮件由东海银行系统自动发送，请勿回复"
- **时间戳**：2041-03-15 09:22

### 关键信息（玩家应提取的）

| 信息 | context 价值 |
|------|-------------|
| 钓鱼域名 `donghai-verify.cn` | ★★★ 核心线索 |
| 实际发件地址 `noreply@donghai-verify.cn` | ★★ 确认域名 |
| 发送时间 2041-03-15 09:22 | ★★ 时间锚点 |
| "24小时内验证否则冻结"的恐吓话术 | ★ 攻击手法特征 |
| 伪造的发件人 `service@donghai-bank.com` | ★ 社工手法 |

### 噪音（不影响推理的内容）

- 邮件头中的 SMTP 路由信息（技术细节，对本案追踪无用）
- 仿冒的银行落款全文
- "请勿回复"等套话

### 推理触发词

当推理输出中出现 **`donghai-verify.cn`**（或该域名的任何变体提及），系统判定为成功推理，解锁下一份证据。

### 理想推理输出（context 包含足够关键信息时）

```
案件: CASE-2041-0001
分析阶段: 钓鱼邮件溯源

一、邮件分析
  该钓鱼邮件使用了典型的紧迫感社工策略（"24小时内验证"）。
  显示发件人 service@donghai-bank.com 是伪造的，
  实际发送自 noreply@donghai-verify.cn。

二、关键发现
  钓鱼链接指向域名 donghai-verify.cn，
  这是攻击者注册的仿冒域名，不属于东海银行官方资产。

三、下一步
  需要查询 donghai-verify.cn 的域名注册信息（WHOIS），
  以确定注册人身份。
```

### 弱推理输出（context 缺少域名信息）

```
案件: CASE-2041-0001
分析阶段: 钓鱼邮件溯源

一、邮件分析
  该邮件使用恐吓语气诱导用户点击链接，
  属于典型的凭证钓鱼攻击。

二、关键发现
  [数据不足] 未能提取到具体的钓鱼域名或技术指标。
  无法确定攻击基础设施。

三、建议
  请补充邮件中的链接地址或发件人详细信息。
```

### ⚠️ Context 管理转折点

**从这一步开始，大多数玩家的 context 会接近满载。** 玩家已经积累了 6 份证据的摘要和 5 次推理输出。如果没有主动清理，context 容量可能已经到了 70%–80%。

**此刻玩家面临的选择：** 是把旧证据全留着（但塞不进新的了），还是开始删掉早期的原始证据 block，只保留推理输出里的结论？这就是 context curation 的进阶——玩家开始做"信息压缩"，虽然他们不会意识到自己在做 compaction。

### 推理触发词

输出中出现 **`刘哲`** 且同时出现 **`受害`** 或 **`客户名单`** 或 **`受影响`** 的提及 → 解锁 Step 07。

**注意：** 这是全案最巧妙的解锁条件。推理引擎需要在看到刘哲的公司信息后，产生"这个人会不会也出现在受害者名单里"的推断。这要求玩家的 context 里同时保留了 briefing_001.eml 中关于"受影响客户名单"的信息和刘哲的身份信息。如果玩家删掉了 briefing 的内容，推理可能不会产生这个关联——玩家需要回头翻 briefing 重新塞关键句进 context。

```

  瑞泽信息技术有限公司由刘哲独资设立于 2041-01-05。
  注册资本极低、实缴为零、未提交年报、使用共享办公地址——
```

### 设计要点

这个发现不依赖任何技术知识——只需要"在名单里认出一个之前见过的名字"。
