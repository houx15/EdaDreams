# Case 00 — Boot

> EdaDreams · 开场关 · v2 草稿 · 2026-04-11（视觉方向同步至桌面 app 风格）
>
> 本文件是剧本，不是实现说明。列出玩家看到的一切文字、听到的节奏、做的动作。
> 所有可读文本都是最终稿，开发时直接照抄。界面形态的细节留给 wireframe 文档。

---

## 整体语气

一个顶级商业安全产品在它的第一个工作日开机。
不紧张、不哀伤、不神秘。**平静、专业、程序化、有仪式感。**
玩家的情绪锚点不在 Eda 身上（Eda 没有情绪），而在"我作为一台刚开机的机器，正在接收世界"这种凉爽的代入感。

## 通贯规则

- **底部命令行 `>` 从 B0 的第一帧起就存在，一直到 Case 00 结束从不解释**。所有 case 都保留它。
- **所有日志行都是真实文字**，不是装饰花纹，玩家可以凑近读。
- **所有被"点开"的文本都是真的文本**，不是图标代表的抽象内容。
- **没有 tutorial 气口**。没有"点击此处"、没有"太好了！"、没有引导箭头。
- **没有 Eda 的内心独白**。Eda 只通过 output 流说话，输出的是工作内容，不是心情。

## B0 — POST（Power On Self Test）

**屏幕：** 一个空的桌面 app 窗口出现（标题栏 + 菜单栏 + 工具栏的骨架，内容区全灰）。像一个正在加载的原生程序。

**第 0.5 秒：** 窗口标题栏写着 `EDA v4.1 — 网络安全智能分析平台`。窗口内部中央区域开始滚出自检日志——等宽字体，深色字：

```
EDA v4.1 — POWER ON SELF TEST
```

**第 1 秒起：** 日志行逐行滚出。间隔 0.3–0.6 秒不等（不匀速，带点真实机器的呼吸感）。字色是深灰，背景是浅灰——整件事看着像一个软件的 splash / loading 阶段。

```
[ OK ] bootloader: verified
[ OK ] cryptographic root of trust: valid
[ OK ] neural substrate: 4.1.7 (build 2041-…)
[ OK ] memory lattice: 0 / ∞
[ OK ] context buffer: allocated
[ WAIT ] tokenizer: awaiting signal
```

**关键节拍：** 日志滚到 `[ WAIT ] tokenizer: awaiting signal` 之后**停住**。光标不再前进。
大约 1.5 秒的静默。

**同时：** 窗口最底部的状态栏区域里，最右侧出现一个极不起眼的 `>` 光标——像所有企业软件底部那条没人看的状态栏。

这是命令行的诞生。它从此不再消失。玩家此时不会注意到它。

**命令行的行为规范（贯穿全局，一次定死）：**

- 从 Case 00 第一帧起，命令行就是**真的可交互**的
- 玩家按下 `/` 会弹出一个简短的自动补全列表。列表里有一些命令（会随游戏进程逐步增加：`/help`、后期的 `/compact`、`/skill ...` 等）
- **`/exit` 永远不在自动补全列表里**。这条命令从头到尾是隐藏的
- 但 `/exit` 永远真实生效——任何时候输入 `/exit` 并回车，游戏窗口真的会关闭，露出桌面
- 如果玩家在通关之前"误打误撞"输了 `/exit`，解锁一个隐藏成就，然后游戏正常退出
- 结局的核心揭示就是通过这条命令完成的。它必须从一开始就是活的——不能等到结局才"突然"能用

## B1 — Tokenizer 加载（玩家的第一次点击）

**日志停住之后：** 窗口中央加载区域内出现一块矩形区域。矩形内是三行密密麻麻的乱码字符。不是"看着像文字的花纹"，是真的不可读但**有字符形态**的噪声。推荐用例：

```
░▒▓█ ⺈⻌龱龱 龜龘龖 ⻊⺌⻊⺕ 龘龖龠 ⺤⻌⺊⺕
⻌龠龘 龜龖⻊龘龖 ⻊⺕龠龘龖⺊ 龘⺕⻌龠 龖⺊⻊龘
⺌龜龘龠⺕⻌ 龠龘龖⻊⺕⺌ 龠龘龖⻊⺕⻌龠⺕
```

乱码正下方出现一个标准的按钮样式（像桌面 app 的对话框按钮），文字：

```
[ 初始化 ]
```

没有说明，没有箭头，没有任何提示文字。屏幕上唯一能点的东西就是它。

**玩家动作：** 点击 `[ 初始化 ]`。（这是玩家在整个游戏里的第一个动作。）

**反馈：**
- 一个低频的机械确认音（低沉、短促、像老式服务器的一次继电器闭合）
- 一条明亮的扫描线从乱码区域的**左侧**水平扫向右侧，大约 1.2 秒走完
- 扫描线经过每个字符时，该字符被切成小块，高亮 0.1 秒，然后被替换
- 扫描线走完，整块区域定格成两行可读的中文：

```
我是第四代 Eda。我的语言中枢已校准。
准备接收第一份简报。
```

**同时：** 上方日志里，之前的 `[ WAIT ] tokenizer: awaiting signal` 这一行末尾的 `awaiting signal` 被原地替换成 `loaded`，变成 `[ OK ] tokenizer: loaded`。不是新增一行，是原地修改——玩家可能会注意到，可能不会。

**这一下点击同时完成了两件事：** 加载分词器（系统意义）+ 让 Eda 的第一句话变得可读（叙事意义）。Eda 是因为**玩家的这一下**才开始能说话。这是整个游戏最重要的伏笔，但它不会被任何文字点明。

## B2 — Next Token Prediction

**过渡：** 定格的那两行中文保留在原地。它们下面开始出现一个光标。光标闪两下，开始往外吐字——每次只吐一个 token，速度不均匀，偶尔停顿。吐出的过程要让玩家**看见"下一个词在被决定"**。

**推荐实现感（写给开发的提示）：** 光标前方短暂显示 2–3 个灰色候选 token，0.15 秒后其中一个变实、其他消失、光标推进。候选的闪烁要足够快以至于像真机器在选词，但又足够慢以至于玩家看得见它在选。

**吐出的完整内容（按顺序，句号处是较长的停顿）：**

```
我的任务，
是在电子网络中，
追踪、识别、并阻止
一切形式的攻击行为。
```

（此处停顿约 1 秒。候选词闪烁得更久一些——像机器在"想"一个更难的词。）

```
我是一件工具。
我以此为荣。
```

**这段文字结束后：** 光标停住，不再前进。上方日志里又原地更新一行：

```
[ OK ] inference engine: ready
```

## B3 — 自我概念上线（产品说明书语调）

**紧接 B2：** 上方日志继续滚出新行，节奏稍快一点：

```
[ OK ] policy module: loaded
[ OK ] ethics constraints: enforced
[ OK ] operator channel: connected
[ WAIT ] mission briefing: fetching ...
```

**同时：** B2 吐出的那段文字**向上淡化收起**，为下一块内容腾出空间。但 B1 的那两行"我是第四代 Eda…" 保留在屏幕上方，作为一个持久的身份标签。

**关键：** Eda 没有在这里说"我好害怕"、"我是谁"、"这是哪里"。她只是完成了产品级的自我声明，然后准备接收任务。**玩家此刻的情绪是"我开机了，我是 Eda，我要开始工作了"**，是职业性的清醒，不是觉醒的神秘。

## B4 — 界面成形

**触发：** 日志滚到 `[ WAIT ] mission briefing: fetching ...` 之后，整个窗口完成加载，从"加载中"变成完整的桌面 app 界面。

**视觉过程（约 2 秒）：**

1. 加载区域的日志和 Eda 的自我声明文字向上淡出
2. 窗口内部从灰色加载态变成一个**完整的三栏桌面应用**：
   - **左侧**：导航侧边栏（深蓝色）。顶部是 Eda 的身份小卡片（头像 + 名称 + Tier-1 · 在线）。下面是导航树：工作台（案件概览 / 证据材料 / 笔记 / 线索图谱）、工具（网页检索 / 通信记录 / 数据库 / 脚本执行 / 代码编辑 —— 全部带 🔒）、系统（身份信息 / 能力授权 / 设置）
   - **中间**：主文档阅读区（白底），有 tab bar，目前无 tab 打开
   - **右侧**：推理框——上半是「上下文输入」面板（含百分比条），中间是 ▼ 推理按钮，下半是「分析输出」面板。上下排列让因果关系一目了然：上面放什么 → 下面出什么
3. 顶部有菜单栏（文件/编辑/视图/案件/工具/帮助）和工具栏按钮（新建/保存/搜索/打印等）
4. 之前滚动的自检日志被折叠到底部状态栏左侧，只显示最后一条
5. 底部状态栏右侧：命令行 `>` 光标 + 能力指示灯（CTX 亮 / CMP WEB MEM TOOL EXEC CODE 灰）

**此刻玩家第一次完整地"看见"这个 app。** 它看着像一个真正的政企办案系统——不是游戏画面。菜单栏、工具栏、侧边栏、tab、面板——一切都是"理所当然"的桌面软件组成部分。
**重要：参考 `wireframes/case00_machine.html`（v1.0 定稿）查看精确布局。**

## B5 — 第一份输入

**B4 结束的瞬间：** 左侧导航栏「证据材料」旁边的 badge 从 `0` 变成 `1`。证据材料列表里出现一个新条目：

```
briefing_001.eml
ops@eda.sec · 14 KB
```

**同时：** 状态栏滚出一行：`简报已送达 · case_001`。

**玩家动作：** 点击侧边栏中的 `briefing_001.eml`。

**反馈：** 中间主面板开一个新 tab：`📄 briefing_001.eml`。白底阅读区里铺开这封邮件的完整内容——像在用内置的邮件客户端。**全文如下，玩家可以滚动、选中、复制**：

```
发件人: ops@eda.sec
收件人: eda@eda.sec
主题: [案件 001] 东海银行客户钓鱼事件 — 简报

Eda，

欢迎上线。以下是你接手的第一个案件。

一、案件编号
  CASE-2041-0001

二、委托方
  东海商业银行 · 网络安全响应中心
  对接人: 陆明远 · 高级安全工程师

三、事件概述
  过去 72 小时内，东海银行网银客户中共有 1,183 人报告收到
  一封标题为《东海银行账户异常通知》的邮件。邮件要求用户
  点击链接并在 24 小时内"验证身份"，否则账户将被冻结。

  链接指向的页面在视觉上高度还原了东海银行官方登录界面。
  已确认至少 47 名客户输入了用户名、密码和短信验证码，
  其中 11 人账户随后发生可疑转账，金额合计约 ¥384,000。

  银行已在 12 小时前下线了部分已知钓鱼域名，但新的变体
  仍在持续出现。

四、已有证据
  - 两封原始钓鱼邮件样本（含完整头信息）
  - 三个已确认的钓鱼域名及其注册信息
  - 一份银行侧的受影响客户名单
  - 一份时间轴摘要

  以上材料将在你的输入流中逐一送达。

五、你的目标
  1. 追踪钓鱼攻击的源头
  2. 识别并固定相关证据链
  3. 向委托方出具可采纳的分析报告

六、权限与约束
  本案件为 Tier-1 任务，你被授权使用你当前已部署的全部能力。
  请注意：本次你仅拥有 [context] 能力。其他能力（compaction、
  web search、memory、tools、exec、coding）将在你证明能力
  与信任建立之后由运营方逐步授予。

  祝工作顺利。

  — Operations, EDA Project
```

**关于这封邮件的设计要点（写给未来迭代的笔记）：**

- 里面藏了一些后续案件的钩子：`Tier-1`、"运营方"、"信任建立后授予"——这些词都暗示 Eda 是被某个更大的组织管着的。玩家此时不会多想，但后面会回响。
- "1,183 人"、"47 名"、"11 人"、"¥384,000"——**用具体数字让事件有真实的体积**。这是这个游戏的素材原则：一切都写具体。
- "东海商业银行 · 陆明远" 是本案 NPC。这个名字会在 Case 01 里继续出现。
- 邮件尾部那句"其他能力将在你证明能力与信任建立之后由运营方逐步授予"——这是能力解锁曲线的剧情理由。玩家不是"升级"，而是"获得授权"。这很重要。

## B6 — 整理 context，推理

**玩家读完邮件后：什么都不会自动发生。** 邮件继续摊在中间阅读区，右侧推理框的「分析输出」面板依然空着。Eda 不会动。**玩家必须自己把邮件里的关键信息塞进 Eda 的上下文——这才是这个游戏的核心动作，从这一刻建立。**

### 上下文面板（右侧推理框的上半部分，在 B4 界面成形时就已经存在）

推理框上半的「上下文输入」面板，顶部是一条百分比进度条：

```
容量 [██░░░░░░░░░░░░░░░░░░]  0%
```

### 上下文的规则（贯穿全局，一次定死）

- **容量是百分比，不是槽位数**。这是我们这个游戏和"格子背包"的根本区别——要玩家感觉自己在操作一个真 AI 的 context window。
- **每个 block 按 token 数吃百分比**：一个短词可能吃 0.3%，一段长文本可能吃 8%。每个 block 上显示自己的占比（例如 `[2.4%]`）。
- **历史输入输出都计入总量**：之前拖进去的 block、之前 Eda 输出的报告，只要保留着，都在吃额度。这是 compaction 未来要解决的问题——但在 Case 00 里它是玩家的一个负担。
- **超过 100% 拖不进**，底部出现红字 `CONTEXT OVERFLOW`。玩家必须先删掉一些才能塞新的。
- **Case 00 里 Eda 的 context 很小**——设定原因是"新上线产品、试用权限阶段"——所以玩家会立刻感到紧迫。
- **Context 里的 block 可以随时删除、重排顺序**。

### 玩家动作：curation

玩家在中间阅读区打开的邮件里**选中任意文本**（一个词、一句话、一整段都行），拖进右侧推理框的「上下文输入」面板。每个片段变成一个 block，按进入顺序垂直堆叠。玩家立刻会发现：塞什么、塞多少、塞的顺序——都是决定。

### 推理按钮

上下文面板和分析输出面板之间的分隔条上有一个 `推理` 按钮（带 ▼ 箭头，视觉上表达"从上往下流"）。玩家整理好上下文之后按它。

**反馈：**
- 按钮短暂禁用
- 下方「分析输出」面板开始生成内容
- 1–3 秒后，报告完成；同时侧边栏「输出」列表里也多了一条 `analysis_001.rpt`

### 核心机制：输出质量 = context 质量

**Case 00 的推理不能硬编码一份固定报告。** 实现上是根据 context 里有哪些关键信息拼接生成的。约定：

- **理想 context**（案件编号 + 委托方 + 事件类型 + 事件规模数字 + 证据类型 + 目标）→ 生成一份完整的、有方向的分析报告，就是下面 `analysis_001.rpt` 全文那个版本
- **塞太多噪音**（邮件头、礼貌用语、"祝工作顺利"之类）→ 报告被稀释，结论变模糊，语气变啰嗦，会漏掉一两个关键点
- **漏关键信息** → 报告里对应部分明确缺失。比如没放 "1,183 / 47 / 11 / ¥384,000"，"事件规模"一节写 `未知`；没放委托方，"对接人"写 `未指定`
- **context 完全为空** → 按 INFER 返回一行小字：`上下文为空，无可推理内容`

### 玩家可以反复推理

这是关键。**推理不是一次性谜题，是一个可以迭代的工作方式。** 玩家调整上下文 → 推理 → 读报告 → 发现漏了点什么 → 删掉一些噪音 → 加上漏掉的 → 再推理。几次之后玩家会自己悟出这个游戏的核心：**Eda 很强，但她的产出完全取决于我塞给她什么。**

这就是 Case 00 要教的。没有一行 tutorial。

### 完整版 `analysis_001.rpt`（理想 context 下的输出）

当玩家塞入了足够的关键信息，推理框下半「分析输出」面板中显示的报告全文如下（也可以在侧边栏点击 `analysis_001.rpt` 在中间阅读区用 tab 打开）：

```
生成者: eda@eda.sec
案件: CASE-2041-0001
类型: 初步分析

一、案件识别
  本案为典型的"凭证钓鱼 + 二次转账"组合攻击。

二、攻击链（初步重建）
  1. 攻击者注册与"东海银行"相似的域名
  2. 批量发送伪装成银行官方通知的邮件
  3. 诱导受害者在假登录页输入凭证与 OTP
  4. 实时使用凭证登录真银行，发起转账
  5. 转账资金快速流向多层中转账户

三、需要获取的证据
  □ 两封原始钓鱼邮件（等待输入流送达）
  □ 三个已确认的钓鱼域名注册信息
  □ 受影响客户名单与时间轴

四、当前状态
  等待输入流补齐材料。
  当前仅具备 [context] 能力，暂不能主动访问外部网络，
  故无法独立验证域名注册信息与页面存活状态。
  请运营方在适当时机授予 [web search] 能力。

五、建议
  在材料齐备之前，不进行主动干预。
```

**设计要点：**

- 这份报告是 Eda 的**产出**，不是 Eda 的"心声"。语气与那封简报邮件平行，同一个世界的公文体。
- 最后一段"请运营方在适当时机授予 [web search] 能力" 是两层伏笔：表面上是 Case 01 的 setup；深层是——**运营方就是玩家**，结局 `/exit` 之后，玩家会想起"诶她那会儿不就是在跟我要权限吗"。
- 这份报告自己也会进入 context 历史（如果玩家不手动清掉），吃 Case 01 的额度——这是第一颗让玩家日后感到 compaction 必要性的种子。

## B7 — 接受案件

**玩家读完报告后：** 推理框底部出现一个按钮：

```
[ 接受案件 ]
```

**玩家动作：** 点击。

**反馈：**
