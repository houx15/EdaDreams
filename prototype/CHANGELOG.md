# EdaDreams 变更日志

## 2026-04-14 — v2: 被动投递 → 主动探索

### 变更原因

v1 的 Case 01 设计中，所有证据文件由系统在推理触发后自动投递到侧边栏。玩家的体验是"有人在给我递信息，我只需要做笔记"——缺乏探索感和主动性。

### 核心变更

| 项目 | v1（旧） | v2（新） |
|------|---------|---------|
| **证据获取** | 推理触发后系统自动投递到侧边栏 | 玩家自己去工具里搜索/查询/打电话获取 |
| **工具交互** | 侧边栏列出工具名但全部锁着 | 网页类工具需要找网址和账号登录；电话拨号联络；数据库需授权 |
| **联络方式** | 即时通讯/聊天框 | 电话拨号（文字化的语音通话） |
| **推理引擎作用** | 输出含触发词 → 解锁下一份文件 | 输出整理已有信息 + 建议调查方向（不投递文件） |
| **调查流程** | 线性 12 步链（Step 01→02→...→12） | 非线性节点图（多条路径可到达终点） |
| **搜索体验** | 无搜索，文件直接出现 | 有噪音结果、无关结果、空结果 |
| **工具发现** | 侧边栏直接显示 | 从 briefing 附件（调查资源备忘）中获取网址和账号 |

### 受影响的文件

| 文件 | 变更程度 | 说明 |
|------|---------|------|
| `script/case01_investigation.md` | **重写** | 从线性 12 步变为节点图式探索 |
| `systems/tools_spec.md` | **大幅修改** | 工具分类改为网页/电话/数据库三类 |
| `systems/inference_engine_spec.md` | 需要更新 | 触发词解锁机制 → 方向建议机制 |
| `content/case01/01–11` | **不变** | 证据内容不变，只是获取方式变了 |
| `content/case01/00_investigation_resources.note` | **新增** | briefing 附件，包含工具网址和账号 |

### 对开发的影响

**需要新增的系统：**
1. **浏览器模拟** — 工具栏地址栏 + 能打开"网页"tab（域名查询/工商/号码查询都是网页）
2. **电话系统** — 拨号界面 + 文字化通话 + NPC 关键词匹配对话
3. **搜索结果引擎** — 每个网页工具有预设的查询→结果映射表，含噪音结果
4. **授权流程** — 玩家通过电话申请 → NPC 确认 → 数据库工具解锁

**可以简化/删除的系统：**
1. ~~自动投递系统~~（不再需要根据推理输出触发文件投递）
2. ~~触发词检测~~（推理输出不再需要包含特定词来解锁文件）

**不受影响的系统：**
1. Context 面板（拖拽、百分比、block 管理 — 全部不变）
2. 推理引擎核心（关键词匹配 → 输出模板拼接 — 不变，只是输出模板需要更新）
3. Boot 序列（Case 00 完全不受影响）

### 设计理念说明

这个变更让 Case 01 从"推理驱动的线性链"变成了"探索驱动的调查游戏"。玩家的核心动作从"在一封封文件里做笔记"变成了"在真实的工具里搜索、打电话要权限、在数据库里翻记录"。

这同时让 delegation 机制更有说服力：Case 01 中玩家亲手做的每一个搜索、每一通电话、每一次数据库查询，都是后续 case 中 Eda 将替他们自动完成的事情。手动做过，才知道自动化有多爽。

## 2026-04-14 — v3: 刘哲是人头 + AutoAI + 真犯林奕辰

### 变更原因

原设计中刘哲就是犯人，案件在追踪到他时基本结束。太快、太简单。
新设计中刘哲是无辜的中间人（外卖骑手被招募），真犯是通过 AutoAI app 远程操控一切的林奕辰（武汉）。

### 核心变更

- **刘哲从主犯变为人头。** 外卖骑手，收 ¥3,000 帮人注册公司/域名，手机被 AutoAI 远程操控执行所有技术操作。
- **陈芳从同伙变为卡农。** 被"跨境代购资金周转"话术招募取现。
- **新增真犯林奕辰。** 武汉洪山区，网络安全研究员转黑，通过 AutoAI 遥控刘哲手机。
- **新增 AutoAI 设定。** 伪装成 AI 效率工具的远程控制 app。Case 01 首次出现，后续 case 反复出现，是贯穿全游戏的暗线。
- **定位真犯靠截图泄露。** AutoAI 日志中一张控制端截图泄露了外卖收货地址，交叉 VPS 的武汉 IP 验证。
- **案件从 ~15 步增加到 ~20 步。** 增加了反转阶段和追踪真凶阶段。
- **两次破案感。** 第一次（以为是刘哲）是假的，第二次（林奕辰）是真的。

### 新增文件

`12_vps_login_records.txt`、`13_phone_forensics.txt`、`14_autoai_screenshot.txt`、`15_ip_lookup.txt`、`16_resident_info.txt`

详见 `script/case01_narrative_v3.md`。

### 2026-04-14 v3 补充：授权链条 + 刘哲被利用细节

- **新增授权链条。** 手机取证需要先用 VPS 双 IP 证据说服陆明远"刘哲不是真正操作者"才能申请。没有 VPS 证据直接要求取证会被拒绝。
- **新增 AutoAI 聊天记录全文。** 完整展示刘哲被招募的过程：从"帮注册公司赚 3000"到手机被控制到被威胁，全部通过 AutoAI 内置聊天进行。
- **调取陈芳通信记录也需要授权链。** 玩家需先用刘哲的证据链说明"陈芳可能也是被利用的"，陆明远才会帮忙调取。
- **陈芳调查为可选支线。** 非结案必须，但给追求完整的玩家额外满足感。

## 2026-04-14 — v2.3: briefing 只给数字 + 交接笔记取代工具清单

### 变更原因

v2.2 给 briefing 附了工具备忘。但现实中领导只给案件信息，不附工具使用指南。
同时，briefing 里直接列出了三个钓鱼域名——但域名应该是玩家自己从钓鱼邮件样本里发现的。

### 变更内容

- **Briefing 删除域名列表。** 不再写"已确认钓鱼域名: donghai-verify.cn..."。只说"附件中包含一封原始钓鱼邮件样本"。玩家打开邮件样本，自己在正文链接和邮件头里发现域名——这是全案第一个"调查动作"。
- **删除 `00_investigation_resources.note`。** 不再给工具清单。
- **新增 `00_handover_note.txt`。** 这是前任 Eda 退役前 + 维护组留下的交接笔记，在系统「备忘」区。写着"怎么查域名用 whois.cnnic-query.cn"、"工商查询用 gsxt.gov.cn"之类的实用信息。语气随意，像老同事留的便条。不是官方文档。
- **流程图新增 Phase 0。** 在域名追踪之前，新增"发现线索"阶段：玩家打开钓鱼邮件附件 → 在邮件里找到域名。

### 设计意图

三层信息发现：briefing 只给数字和背景 → 钓鱼邮件样本里藏着第一条线索（域名）→ 交接笔记告诉你工具怎么用。玩家需要自己把三层串起来。

## 2026-04-14 — v2.2: 删除工具备忘，所有工具通过搜索引擎发现

### 变更原因

v2 给了玩家一份《调查资源备忘》列出所有可用网址和账号。但现实中领导只会给你一份 briefing，不会附一张工具清单。工具应该由玩家自己通过搜索引擎发现。

### 变更内容

- **删除 `00_investigation_resources.note`。** 不再给玩家任何工具清单。
- **Briefing 邮件修改：** 删除"已有证据将逐一送达"的承诺。只给案件信息 + 三个域名 + 一份钓鱼邮件样本附件。对接人陆明远只写了名字和职位，**没有电话号码**。
- **搜索引擎成为唯一入口。** 域名查询、工商信息、号码查询等网站全部通过搜索引擎发现。陆明远的联系方式也要搜。
- **侧边栏工具区只保留「网页搜索」一个入口。**

### 对开发的影响

搜索引擎结果库需要扩充：除了案件相关搜索外，还需要预设"域名查询"、"WHOIS"、"工商查询"、"号码归属"、"东海银行 陆明远"等工具发现类搜索的结果。

## 2026-04-14 — v2.1: 证据文件"去编辑化"

### 变更原因

v2 改了获取方式但没改内容。证据文件仍然像有人帮玩家整理好了——有分析备注、有"异常标记"、甚至有现成的资金流向分析和时间轴。但这些整理和分析工作是玩家的任务，不应该有人替他们做好。

### 变更内容

| 文件 | 变更 |
|------|------|
| `07_affected_clients.txt` | **删除底部"异常标记"段落。** 银行系统不会帮玩家标出"刘哲很可疑"。玩家需要自己在 47 个名字里发现刘哲。这是全案最重要的啊哈时刻。 |
| `08_liuzhe_account.txt` | **删除"过水模式"分析备注。** 银行只输出原始流水数据。"资金停留时间短"这个判断由玩家自己做出。 |
| `02_domain_registration.txt` | **拆分为逐域名查询结果。** 玩家每次输入一个域名，得到一条结果。不再一次性返回三个域名的汇总。增加 donghai-bank.com（正规域名）供对比。 |
| `03_email_account.txt` | **删除"该邮箱是攻击邮件的实际发送源"分析注释。** |
| `04_phone_records.txt` | **删除"疑似批量发送"、"异常偏高"等分析性备注。** 通信记录只是原始数据。 |
| `06_company_registration.txt` | **删除"该类型地址通常用于仅需工商注册而无实际办公需求的场景"的分析。** 工商系统不帮你判断是不是壳公司。 |
| `10_account_b.txt` | **删除"开户审核记录"中柜员备注**（"近期有一笔亲属转账需要接收"这条线索太直白，等于告诉玩家这个账户是预谋的）。 |
| `09_fund_flow.txt` | **改为推理输出模板（重命名为 `09_fund_flow_INFERENCE_OUTPUT.txt`）。** 资金流向分析是 Eda 推理的产物，不是预存的证据。玩家把流水数据放进 context 推理后，Eda 生成这份分析。 |
| `11_timeline.txt` | **改为推理输出模板（重命名为 `11_timeline_INFERENCE_OUTPUT.txt`）。** 时间轴是 Eda 综合全案信息后生成的，不是外部提供的。 |

### 设计原则

**证据文件 = 原始数据源的输出。** 它们只包含该数据源能提供的信息，不包含任何分析、判断或交叉比对。所有分析工作由玩家+Eda（推理引擎）完成。

**判断哪些信息保留的标准：** "这个信息是这个系统/数据库本身就会显示的吗？" 是 → 保留。否 → 删除或改为推理输出。
